1. Overview

GlucoLog is a client-side only, single-file HTML application designed for women with diabetes to track daily food intake, blood glucose readings, and generate medical reports for healthcare providers.

Key Constraints That Drove Every Decision
  • Must deploy on GitHub Pages (static hosting, no server)
  • Must be a single HTML file (no build step, no bundler)
  • Must work offline-capable after first load
  • Must look professional and medical, not like a toy
  • Target user: non-technical person on a phone

Tech Stack

Layer Technology Why This Choice
Markup HTML5 Semantic elements, native form validation, <input type="date"> for date picker
Styling CSS3 Custom properties (design tokens), Flexbox/Grid, no framework overhead
Logic Vanilla JS Zero dependencies, no build step, ~400 lines total
Storage localStorage Persistent across sessions, no backend needed, 5-10MB quota
Hosting GitHub Pages Free, automatic deployment via Actions, HTTPS included

2. Why Single-File HTML

Advantages

  • Zero build pipeline. No webpack, vite, rollup, npm, or node_modules. Edit the file, push, done.
  • Instant deployment. GitHub Pages serves the file directly. No CI compilation step needed (the workflow just uploads the file as-is).
  • Offline-friendly. Once loaded, the browser caches the single file. All logic and data are local.
  • Easy to fork and modify. Any developer can open one file and see everything.
  • No CORS issues. No API calls, no cross-origin requests, no tokens.

Trade-offs Accepted

  • No code splitting. The entire app loads at once (~45KB). Acceptable for this app size.
  • No TypeScript. We lose type safety but gain zero-config simplicity.
  • CSS and JS are not minified. Compression by the server (gzip) makes this negligible.
  • No component framework. DOM manipulation is done via innerHTML string templates. This works for a small app but would not scale to 50+ interactive components.
  • No data sync. Data lives only on the user's device. Clearing browser data deletes everything.
When to migrate away from single-file If the app grows beyond ~2,000 lines, or if you need user accounts, cloud sync, or a nutritional database API, migrate to a framework (e.g., React/Next.js or SvelteKit) with a proper backend.

The architecture follows a three-layer pattern:

  1. Presentation Layer — HTML structure + CSS. Pure visual layout, no logic.
  2. Logic Layer — JavaScript functions organized by domain (food, glucose, reports). Handles user actions and calls the rendering engine.
  3. Data Layer — Two functions (getData(), saveData()) that abstract all localStorage access behind a single JSON object.

5. Data Layer

Data Schema

All application state lives in a single localStorage key: glucolog_data. The value is a JSON object with three top-level namespaces:

{
  "entries": {
    "2026-03-20": [
      {
        "id": 1710936000000,      // Date.now() at creation - unique ID
        "name": "Oatmeal (1 cup)", // Free-text food name
        "meal": "Breakfast",       // Enum: Breakfast|Lunch|Dinner|Snack
        "calories": 150,           // kcal (integer)
        "carbs": 27,               // grams (float)
        "sugar": 1,                // grams (float)
        "fiber": 4,                // grams (float)
        "protein": 5,              // grams (float)
        "portion": "1 cup",        // Free-text portion description
        "notes": "",               // Free-text user notes
        "time": "08:30 AM"         // Display time (from toLocaleTimeString)
      }
    ]
  },
  "glucose": {
    "2026-03-20": [
      {
        "id": 1710936060000,       // Date.now() unique ID
        "value": 112,              // mg/dL (integer, range 30-600)
        "context": "fasting",      // Enum: fasting|before-meal|after-meal-1h|
                                   //       after-meal-2h|bedtime
        "time": "07:15 AM"         // Display time
      }
    ]
  },
  "settings": {
    "patientName": "Jane Doe",     // Persisted for report generation
    "doctorName": "Dr. Smith",     // Persisted for report generation
    "diabetesType": "Type 2"       // Enum: Type 1|Type 2|Gestational|Prediabetes
  }
}

Storage Strategy

Why localStorage (not IndexedDB, not cookies)

OptionVerdictReasoning
localStorageChosenSynchronous API, simple JSON serialization, 5-10MB quota (enough for years of food logs), universal browser support
IndexedDBRejectedAsync API adds complexity. Needed for large binary data or complex queries. Overkill for structured JSON < 1MB.
CookiesRejected4KB limit, sent with every HTTP request, not designed for application state
sessionStorageRejectedCleared when tab closes. Data must persist across sessions.

Read/Write Pattern

// READ: Deserialize entire state from localStorage
function getData() {
    const raw = localStorage.getItem('glucolog_data');
    return raw ? JSON.parse(raw) : { entries: {}, glucose: {}, settings: {} };
}

// WRITE: Serialize entire state back to localStorage
function saveData(data) {
    localStorage.setItem('glucolog_data', JSON.stringify(data));
}
Design Decision: Full Overwrite vs. Granular Updates We read and write the entire JSON object on every operation. This is intentionally simple. For a food tracker with ~10-30 entries per day, the serialization cost is negligible (< 1ms). A granular approach (patching specific keys) would add complexity without measurable benefit at this data scale.

Date Keying

Both entries and glucose use ISO date strings ("2026-03-20") as object keys. This design:

  • Enables O(1) lookup for any date's data — no filtering arrays.
  • Natural partitioning — each day is isolated, no cross-day contamination.
  • String-sortable — ISO dates sort lexicographically, enabling date range iterations in the report engine.
// Date key generation
function getDateKey(date) {
    return date.toISOString().split('T')[0];  // "2026-03-20"
}

// Current view's date key (from the date input element)
function getCurrentDateKey() {
    return document.getElementById('currentDate').value;
}

Responsive Design

The app uses a mobile-first, single-column layout capped at max-width: 600px. This was deliberate: food tracking is primarily a phone activity done at meal times.

/* Base layout - already mobile-optimized */
.main {
    max-width: 600px;
    margin: 0 auto;
    padding: 16px;
}

/* Only breakpoint: small phones (< 480px) */
@media (max-width: 480px) {
    .macro-grid { grid-template-columns: repeat(2, 1fr); }  /* 4-col → 2-col */
    .header-btn span.btn-text { display: none; }             /* Hide button labels */
    .glucose-input-row { flex-wrap: wrap; }                   /* Stack glucose fields */
    .glucose-input-row .field { min-width: calc(50% - 5px); }
}
Why only one breakpoint? The 600px max-width means the app looks identical on tablet and desktop — just centered with whitespace. This is intentional for a medical tool: consistency across devices reduces user confusion. The only adaptation is for very small phones where the 4-column macro grid becomes 2-column.

Food CRUD in Detail

Create (addFoodEntry)

function addFoodEntry() {
    // 1. Validate: name is required
    const name = document.getElementById('foodName').value.trim();
    if (!name) { showToast('Please enter a food name'); return; }

    // 2. Build entry object from form fields
    const entry = {
        id: Date.now(),  // Millisecond timestamp = unique enough for single-user
        name: name,
        meal: document.getElementById('mealType').value,
        calories: parseFloat(document.getElementById('calories').value) || 0,
        carbs: parseFloat(document.getElementById('carbs').value) || 0,
        // ... other macros
        time: new Date().toLocaleTimeString([], { hour: '2-digit', minute: '2-digit' })
    };

    // 3. Insert into date-keyed array
    const dateKey = getCurrentDateKey();
    const data = getData();
    if (!data.entries[dateKey]) data.entries[dateKey] = [];
    data.entries[dateKey].push(entry);
    saveData(data);

    // 4. Clear form + re-render
    // ... reset all input values
    loadDate();         // Triggers renderFoodLog + updateDailySummary
    showToast('Food entry added');
}

Quick Add Pattern

// Quick-add buttons call the same addFoodEntry() but pre-fill the form first
function quickAdd(name, meal, cal, carbs, sugar, fiber, protein) {
    document.getElementById('foodName').value = name;
    document.getElementById('mealType').value = meal;
    document.getElementById('calories').value = cal;
    // ... set other fields
    addFoodEntry();  // Reuse the same create flow
}

// In HTML: onclick="quickAdd('Oatmeal (1 cup)', 'Breakfast', 150, 27, 1, 4, 5)"
Why this approach? Quick-add and manual entry share the same code path. No duplication. The quick-add buttons are just a UX shortcut that pre-fills the form and submits it. This means any validation or logic change in addFoodEntry() automatically applies to quick-adds too.

Delete

function deleteEntry(id) {
    const dateKey = getCurrentDateKey();
    const data = getData();
    // Filter out the entry by its unique timestamp ID
    data.entries[dateKey] = data.entries[dateKey].filter(e => e.id !== id);
    saveData(data);
    loadDate();  // Re-render
}

Rendering (Food Log)

function renderFoodLog() {
    const entries = data.entries[dateKey] || [];

    // 1. Empty state check
    if (entries.length === 0) {
        container.innerHTML = '<div class="empty-state">...</div>';
        return;
    }

    // 2. Group entries by meal type
    const mealOrder = ['Breakfast', 'Lunch', 'Dinner', 'Snack'];
    const grouped = {};
    entries.forEach(e => {
        if (!grouped[e.meal]) grouped[e.meal] = [];
        grouped[e.meal].push(e);
    });

    // 3. Build HTML string (meal group → entries within)
    let html = '';
    mealOrder.forEach(meal => {
        if (!grouped[meal]) return;
        html += '<div class="meal-group">...';
        // Each entry gets its macros displayed + a delete button
    });

    container.innerHTML = html;
}
innerHTML vs. DOM API We use innerHTML with string concatenation rather than createElement/appendChild. This is a conscious trade-off: string templates are far more readable for HTML generation and adequate for our data size. The entire food log is re-rendered on every change (not diffed). For < 100 entries per day, this is imperceptible. A framework like React or Svelte would be warranted at larger scale.

Glucose System

// Clinical classification of glucose readings
function getGlucoseStatus(value) {
    if (value < 70)  return { label: 'Low', cls: 'high' };     // Hypoglycemia → red
    if (value <= 100) return { label: 'Normal', cls: 'normal' }; // Green
    if (value <= 140) return { label: 'Elevated', cls: 'elevated' }; // Amber
    return { label: 'High', cls: 'high' };                        // Hyperglycemia → red
}

// Note: "Low" uses the 'high' CSS class (red) intentionally —
// both hypo and hyperglycemia are dangerous conditions

Report Generation Engine

The report generator is the most complex function (~100 lines). It:

  1. Accepts parameters from the modal form: patient name, doctor name, diabetes type, date range (1-30 days), additional notes.
  2. Iterates over the date range using a for loop with Date objects, collecting food entries and glucose readings per day.
  3. Computes aggregate statistics:
    • Average daily calories, carbs, sugar, fiber, protein
    • Glucose: average, min, max, time-in-range percentage
    • Hyperglycemia count (>180 mg/dL)
    • Hypoglycemia count (<70 mg/dL)
  4. Formats as plain text with monospace-friendly alignment (column widths, separator lines). This ensures the report looks professional when pasted into an email.
  5. Includes a medical disclaimer at the bottom: "Data has not been independently verified."

Share & Email System

Share Modal

Copy to Clipboard

function copyToClipboard(elementId) {
    const text = document.getElementById(elementId).textContent;

    // Modern API (requires HTTPS or localhost)
    navigator.clipboard.writeText(text).then(() => {
        showToast('Copied to clipboard');
    }).catch(() => {
        // Fallback for older browsers or HTTP contexts
        const ta = document.createElement('textarea');
        ta.value = text;
        document.body.appendChild(ta);
        ta.select();
        document.execCommand('copy');
        document.body.removeChild(ta);
        showToast('Copied to clipboard');
    });
}

10. Deployment Pipeline

GitHub Actions Workflow

# .github/workflows/deploy.yml
name: Deploy to GitHub Pages

on:
  push:
    branches: [main]          # Trigger: any push to main
  workflow_dispatch:            # Also allow manual trigger

permissions:
  contents: read               # Read repo files
  pages: write                 # Write to Pages
  id-token: write              # OIDC token for Pages deployment

jobs:
  deploy:
    environment:
      name: github-pages
      url: ${{ steps.deployment.outputs.page_url }}
    runs-on: ubuntu-latest
    steps:
      - uses: actions/checkout@v4           # 1. Clone repo
      - uses: actions/configure-pages@v5    # 2. Configure Pages
      - uses: actions/upload-pages-artifact@v3  # 3. Upload root dir
        with:
          path: .                           #    (entire repo = site)
      - uses: actions/deploy-pages@v4       # 4. Deploy to Pages CDN

11. Security Considerations

XSS Protection

// All user-generated text is escaped before insertion into innerHTML
function escapeHtml(str) {
    const div = document.createElement('div');
    div.textContent = str;      // textContent auto-escapes
    return div.innerHTML;       // Returns escaped HTML entities
}

// Used everywhere user data appears:
// <div class="food-name">${escapeHtml(e.name)}</div>

Without this, a user entering <script>alert('xss')</script> as a food name would execute arbitrary JavaScript. The escapeHtml() function converts < to &lt; and > to &gt;, rendering the input as harmless text.

Data Privacy

  • All data stays on-device. No network requests, no analytics, no cookies, no tracking.
  • No authentication. Anyone with physical access to the device can view the data.
  • localStorage is domain-scoped. Other websites cannot access GlucoLog's data.
  • Email report uses mailto:. The report content passes through the OS email client, not through any server.

Input Validation

  • Glucose values are bounded: min="30" max="600" (validated in JS too)
  • Numeric fields use type="number" with min="0"
  • Food name is required (empty string check in JS)
  • parseFloat() || 0 ensures NaN inputs default to zero, not corrupt data

12. Rebuild Guide (Step-by-Step)

If you need to rebuild this app from scratch (or build a similar one), follow this sequence:

Phase 1: Foundation (Day 1)

  1. Create index.html with doctype, meta viewport, and an empty <style> + <script> block.
  2. Define CSS custom properties in :root. Choose your color palette first — it affects everything. Use a medical blue (#4A6FA5 family) for healthcare trust.
  3. Build the data layer: write getData(), saveData(), getDateKey(). Define your JSON schema (entries keyed by date, each entry as an object with an id timestamp).
  4. Build the card CSS pattern: a white card with border, radius, shadow, and padding. You'll reuse this for every section.

Phase 2: Core CRUD (Day 1-2)

  1. Build the food entry form: inputs for name, meal type, calories, carbs, sugar, fiber, protein, portion, notes.
  2. Implement addFoodEntry(): read form values, build entry object, push to date array, save, clear form.
  3. Implement renderFoodLog(): group entries by meal type, build HTML string with meal group headers + entry rows. Handle empty state.
  4. Implement deleteEntry(id): filter array, re-save, re-render.
  5. Add the daily summary: reduce entries to totals (cal, carbs, sugar, fiber), update DOM elements.

Phase 3: Glucose Tracking (Day 2)

  1. Add glucose input section: mg/dL number input + context dropdown (fasting, before meal, etc.).
  2. Implement logGlucose(): same pattern as food entries but in the glucose namespace.
  3. Implement getGlucoseStatus(): return label + CSS class based on clinical thresholds.
  4. Render glucose readings with color-coded badges.

Phase 4: Date Navigation (Day 2)

  1. Add date input (<input type="date">) and prev/next buttons.
  2. Implement changeDate(delta): adjust date by +/- 1 day, update input value, call loadDate().
  3. Implement loadDate(): calls all three render functions (food log, glucose log, summary).

Phase 5: Export Features (Day 3)

  1. Build modal system: overlay + centered box + sticky header/footer.
  2. Implement share: format current day as plain text, show in modal, copy to clipboard.
  3. Implement report generator: iterate date range, aggregate statistics, format as medical report, add email button with mailto:.

Phase 6: Education Content (Day 3)

  1. Build tabbed panel: horizontal scrollable tab bar + content sections toggled by class.
  2. Add 7 education sections based on ADA guidelines, covering foods, timing, glucose monitoring, women's health.

Phase 7: Polish (Day 4)

  1. Quick-add buttons: common foods with pre-filled macro values.
  2. Toast notification system: fixed-position element, show/hide with CSS transition.
  3. Color-coded warnings: carb/sugar thresholds on daily summary.
  4. Responsive breakpoint: adjust grid and hide button text on small screens.
  5. Print styles: hide interactive elements for printing.
  6. XSS protection: escape all user-generated text before innerHTML.

Phase 8: Deploy (Day 4)

  1. Create .github/workflows/deploy.yml with the Pages deployment workflow.
  2. Enable GitHub Pages in repository settings (Source: GitHub Actions).
  3. Push to main: workflow deploys automatically.
